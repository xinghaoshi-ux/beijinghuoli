# UI 设计简报

## 1. 设计目标

本项目 UI 应服务于赛事查询和后台录入，风格应清晰、克制、易读。

公共端要让参赛者在手机上快速查到结果和排名。后台端要让管理员高效录入和修正数据。

## 2. 公共端设计原则

- 首屏直接展示核心功能，不做营销页。
- 赛事名称清晰可见。
- 搜索框和 tab 入口在首屏可见。
- 三个 tab 的视觉权重一致。
- 排名数字、姓名、球队名、积分要易扫读。
- 手机端优先考虑卡片和紧凑列表。

## 2.1 公共端视觉基准

公共端视觉方向以用户确认的参考图为准：

- 顶部使用大尺寸赛事标题卡片。
- 标题卡片内可使用网球、球拍、浅色曲线作为轻装饰。
- 主标题为深紫黑色。
- 主操作色使用紫色。
- 强调色使用荧光黄绿色。
- 页面背景使用浅紫灰。
- 内容区使用白色大卡片承载。
- 搜索框、tab、表格都使用圆角和轻边框。
- `比赛结果 / 球队排名 / 个人排名` 使用三段式 tab。
- 当前 tab 使用紫色实底，高亮状态可带小三角指示。
- 比赛结果表头使用浅紫底色。
- 积分数字使用绿色强调。
- 可使用少量网球/球拍线稿装饰，但不要影响数据阅读。

手机端不能直接照搬桌面表格，需要改为卡片列表。

## 3. 后台设计原则

- 使用工作台式布局。
- 优先表格、表单、弹窗或抽屉。
- 不使用大面积装饰性 hero。
- 不使用复杂视觉背景。
- 操作按钮清晰，危险操作需要二次确认。

## 4. 公共端页面结构

### 4.1 桌面端

结构建议：

```text
赛事标题
搜索框
tab: 比赛结果 / 球队排名 / 个人排名
内容区域
```

比赛结果：

- 使用表格或分组列表。
- 同一项目下结果可分组展示。

排名：

- 使用表格。
- 排名、名称、积分是主要列。

### 4.2 手机端

结构建议：

```text
赛事标题
搜索框
横向或等宽 tab
卡片列表
```

要求：

- 不横向滚动。
- 排名卡片高度稳定。
- 积分醒目但不过度装饰。
- 长名称允许换行。

## 5. 后台页面结构

后台建议使用：

- 左侧导航
- 顶部栏
- 主内容区

主要页面：

- 概览
- 球员管理
- 球队管理
- 成绩录入
- 成绩管理
- 排名预览

## 6. 组件建议

公共端：

- 搜索输入框
- tab
- 排名卡片
- 结果卡片
- 桌面表格
- 空状态
- 加载状态

后台端：

- 表格
- 表单
- 选择器
- 多选球员控件
- 文件上传
- 解析预览表格
- 弹窗或抽屉
- 确认对话框

## 7. 文案原则

- 文案简短。
- 使用赛事用户能理解的词。
- 不在页面中解释复杂系统设计。
- 空状态应直接说明当前无数据。

示例：

- 暂无比赛结果
- 暂无球队排名
- 暂无个人排名
- 未找到匹配结果
- 上传并解析
- 确认导入

## 8. 颜色与视觉

建议使用参考图中的清爽赛事风格：

- 主色使用紫色。
- 辅助强调色使用荧光黄绿色。
- 背景使用浅紫灰或白色。
- 积分和排名使用清晰强调色。
- 避免大面积深色背景。
- 装饰元素只放在 Header、空状态或卡片角落。

## 9. 可访问性与可用性

- 字体大小在手机端必须可读。
- 点击区域不能过小。
- 表单错误提示必须靠近字段。
- 加载和错误状态必须明确。
- 积分和排名不能只靠颜色表达。

## 10. 明确不做

本版 UI 不做：

- 球员主页
- 球队主页
- 复杂图表
- 大型营销首页
- 动画展示
- 数据导出入口
